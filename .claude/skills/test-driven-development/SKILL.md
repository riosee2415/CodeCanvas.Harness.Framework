# 테스트 주도 개발 (TDD)

> 테스트를 **먼저** 쓴다. 실패를 **본다**. 통과하는 **최소** 코드를 쓴다.
> 핵심: 테스트가 실패하는 걸 못 봤다면, 그 테스트가 옳은 것을 검증하는지 알 수 없다.

## 철의 규칙

```
실패하는 테스트 없이는 프로덕션 코드를 쓰지 않는다.
```

- 테스트보다 코드를 먼저 썼다면 → **지운다**. 참고용으로도 남기지 않는다. 테스트부터 다시 시작.
- 예외(드묾): 일회용 프로토타입·생성코드·설정파일. 애매하면 TDD를 따른다.

## RED → GREEN → REFACTOR

**RED — 실패하는 테스트**
- 한 번에 **한 행동**만. 이름에 "그리고"가 들어가면 쪼갠다.
- 실제 코드를 대상으로 — mock은 불가피할 때만.
- 테스트를 돌려 **실패를 눈으로 확인**하고, *기능이 없어서* 실패하는지(오타·컴파일 에러가 아니라) 본다.
- 바로 통과하면? 이미 있는 동작을 테스트한 것 → 테스트를 고친다.

**GREEN — 최소 구현**
- 통과시키는 **가장 단순한 코드**만. 요청 안 한 기능·옵션·미래 대비 추상화 금지(YAGNI).
- 통과 확인 후 **다른 테스트도 여전히 green**인지 본다. 실패하면 테스트가 아니라 **코드**를 고친다.

**REFACTOR — 정리 (green일 때만)**
- 중복 제거·이름 개선·헬퍼 추출. **새 동작 추가 금지.** 테스트는 계속 green.

→ 다음 행동을 위해 다음 실패 테스트로 반복. 사이클은 작게(1~10분, "baby steps").

## 합리화 신호 (멈춰라)

- "이번 한 번만 건너뛰자" · "나중에 테스트 추가" · "이미 수동으로 해봤어" · "지우긴 아까워(매몰비용)"
  → 전부 TDD를 어기는 신호. 코드를 지우고 테스트부터.

## 막히면

- 어떻게 테스트할지 모르겠다 → 원하는 API를 먼저 적고 거기서 단언(assert)부터.
- 테스트가 너무 복잡 → 설계가 복잡한 것. 인터페이스를 단순화한다.
- 전부 mock해야 함 → 결합도가 높음. 의존성 주입을 고려한다.
