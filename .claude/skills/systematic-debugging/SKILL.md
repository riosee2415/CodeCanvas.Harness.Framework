# 체계적 디버깅

> 고치기 전에 **이해한다**. 증상이 아니라 **근본 원인**을 고친다.

## 절차

1. **재현 먼저**: 무엇이든 바꾸기 전에, 정확한 단계로 버그를 **일관되게 재현**한다. 재현 못 하면 고쳤는지도 모른다.
2. **증거를 끝까지 읽는다**: 전체 에러 메시지·스택트레이스·줄번호. 경고를 건너뛰지 않는다.
3. **최근 변경을 본다**: 최근 커밋·의존성·설정. 데이터 흐름을 **나쁜 값의 출처까지 역추적**한다.
4. **동작하는 기준과 비교**: 잘 되는 레퍼런스를 끝까지 읽고, 모든 차이를 나열한다.
5. **가설 하나를 글로 적는다**: 가장 작은 변경으로, **한 번에 한 변수만** 바꿔 검증한다.
6. **버그를 실패 테스트로 포착**: 버그를 재현하는 테스트를 쓰고 → 고치고 → 다른 게 안 깨지는지 확인한다.
   이 테스트가 회귀를 막는다 (test-driven-development 참조).
7. **3회 실패하면 멈춘다**: 같은 접근이 3번 실패하면 가정·아키텍처를 의심한다. 이해한 척하지 말고 도움을 요청한다.

## 안티패턴 (멈춰라)

- 재현도 안 하고 추측으로 코드를 바꾼다.
- 에러 메시지를 대충 보고 넘긴다.
- 한 번에 여러 곳을 바꿔서 무엇이 고쳤는지 모른다.
- 실패 테스트를 **지우거나 약화**시켜 "고친다" — 원인이 아니라 증상을 숨기는 것. 절대 금지.
