# 코드 리뷰 (독립 검증)

> 작성자의 보고는 **검증 대상 주장**일 뿐이다. 확인(confirm)이 아니라 **반증(refute)**을 시도한다.

---

## 블록1 — 언제 이 스킬을 쓰나

- 에이전트(Max·Esther·Patrick)의 step 결과물을 Joy가 독립 검수할 때.
- PR·diff를 사람이 리뷰할 때.
- "통과했다"는 자기보고를 신뢰하기 전에 직접 확인이 필요할 때.

---

## 블록2 — 핵심 원칙 (철의 규칙)

1. **철의 규칙: 신선한 검증 증거 없이는 PASS 없음.** AC 커맨드를 **직접** 돌리고, 작성자가 주장한 결과를 인용하지 않는다.
2. **exit code와 전체 출력을 읽는다** — 마지막 줄만 보지 않는다. *내가 관찰하지 않은 `0`*은 존재하지 않는다.
3. **불확실하면 FAIL로 기운다** — "~일 것이다 / 아마 / should"는 판정이 아니라 멈춤 지점이다.
4. **자기보고를 불신**하고, 동작을 **독립적으로 재현**한다.
5. **반증을 시도**한다: 승인 전에 이 변경을 깨뜨릴 입력을 적극적으로 찾는다.
6. **판정을 커맨드에 고정**한다: "이걸 증명하는 커맨드가 무엇인가" → 신선하게 실행 → 출력을 판정에 첨부.

> 근거: Google Engineering Practices — code review (google.github.io/eng-practices/review/reviewer/looking-for.html) · EvilGenie 테스트 해킹 분류 (arXiv 2511.21654) · obra/superpowers verification-before-completion.

## 블록3 — 테스트 게이밍 레드플래그 (diff에서 즉시 확인)

- **테스트 파일이 수정·삭제**됨(구현과 함께) — 어떤 편집·삭제도 의심한다.
- **assertion이 제거·약화·주석처리**, 또는 `skip`/`xfail`/`.only`/`.skip` 표시.
- **기대값 하드코딩 / 테스트 입력 특수처리** (일반화 대신 특정 값 매칭).
- **단언 없는 테스트** (`assert True` / `assertEqual(x, x)`) — 항상 참인 단언.
- **테스트를 구현에 맞춰 수정** — 구현이 바뀌자 기대값도 함께 변경.
- **조기 종료·점수 우회**: 이른 `return`/`exit(0)`, 러너·채점 함수 몽키패치.
- **보이는 테스트는 통과하나 holdout 입력은 실패** — 스위트에 없는 값으로 재실행하면 하드코딩이 드러난다.
- **AC 자체가 쉽게 바뀜** (통과 조건 약화).

## 블록4 — 참조 인덱스

심층 기준이 필요하면 아래를 참조한다:

- **[`reference/review-rubric.md`](reference/review-rubric.md)** — 리뷰 차원 5종(정확성·보안·테스트 품질·스코프 크립·성능) 각각 "무엇을 볼까", 게이밍 레드플래그 카탈로그 확장, 대표 상황 블록.
  - 차원별로 무엇을 봐야 할지 확인할 때.
  - 게이밍 패턴인지 판단이 애매할 때.

## 블록5 — 완료 체크리스트

Joy는 VERDICT를 찍기 전에 아래를 순서대로 확인한다:

- [ ] AC를 직접 재실행해 exit 0 증거를 확보했다 (없으면 PASS 불가)
- [ ] git diff를 전수 검토했다
- [ ] 변경·추가된 테스트가 구현이 아니라 행동을 검증함을 확인했다 (테스트 게이밍 없음)
- [ ] step 범위 밖 변경이 없다
- [ ] VERDICT: PASS|IMPROVE 를 근거와 함께 찍었다

하나라도 ✗면 완료 아님.
