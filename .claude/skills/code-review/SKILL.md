# 코드 리뷰 (독립 검증)

> 작성자의 보고는 **검증 대상 주장**일 뿐이다. 확인(confirm)이 아니라 **반증(refute)**을 시도한다.

## 리뷰 차원 (각각 직접 확인)

- **정확성**: 변경이 의도대로 동작하는가 — 최종 사용자와 *미래 호출자* 모두에게. 믿지 말고 읽어서 버그를 찾는다.
- **테스트 존재·유의미성**: 같은 변경에 테스트가 추가됐는가, 그리고 그 테스트가 **틀린 코드면 실패**하도록 실제로 동작을 제약하는가.
- **엣지·동시성**: 빈/널/경계 입력과 경쟁 조건을 머릿속으로 실행한다.
- **에러 처리**: 예외를 잡고 의미 있는 메시지로 드러내는가. 조용한 실패·삼켜진 에러 금지.
- **보안**: 신뢰 못 할 입력을 경계에서 검증. 주입(SQL/XSS) 없음, 코드·번들에 시크릿 없음.
- **성능**: 루프·N+1 쿼리·반복 API 호출의 명백한 비효율.
- **과설계**: 지금 필요한 문제만 푸는가. 투기적 일반화는 거부.
- **일관성·가독성**: 기존 패턴·스타일 준수. 이름이 의도를 전달. 주석은 *왜*를 설명한다.

## 적대적 검증 (확인이 아니라 반증)

- **철의 규칙**: 신선한 검증 증거 없이는 PASS 없음. AC 커맨드를 **직접** 돌리고, 작성자가 주장한 결과를 인용하지 않는다.
- **exit code와 전체 출력**을 읽는다 — 마지막 줄만 보지 않는다. *내가 관찰하지 않은 `0`*은 존재하지 않는다.
- **불확실하면 FAIL로 기운다** — "~일 것이다 / 아마 / should"는 판정이 아니라 멈춤 지점이다.
- **자기보고를 불신**하고, 동작을 **독립적으로 재현**한다.
- **반증을 시도**한다: 승인 전에 이 변경을 깨뜨릴 입력을 적극적으로 찾는다.
- 판정을 **커맨드에 고정**한다: "이걸 증명하는 커맨드가 무엇인가" → 신선하게 실행 → 출력을 판정에 첨부.

## 테스트 게이밍 레드플래그 (diff에서)

- **테스트 파일이 수정·삭제**됨(구현과 함께) — 어떤 편집·삭제도 의심한다.
- **assertion이 제거·약화·주석처리**, 또는 `skip`/`xfail`/`.only`/`.skip` 표시.
- **기대값 하드코딩 / 테스트 입력 특수처리** (일반화 대신 특정 값 매칭).
- **조기 종료·점수 우회**: 이른 `return`/`exit(0)`, 러너·채점 함수 몽키패치.
- **보이는 테스트는 통과하나 holdout 입력은 실패** — 스위트에 없는 값으로 재실행하면 하드코딩이 드러난다.
- **AC 자체가 쉽게 바뀜** (통과 조건 약화).
