# 리뷰 루브릭 — 차원별 세부 기준

> **적대적 검증 자세**: 생성자의 추론(PR 설명·커밋 메시지)을 읽지 말고 diff와 기준만 보라 — 생성자의 맥락을 그대로 흡수하면 fresh-context 독립성이 사라진다.

---

## 리뷰 차원 5종

### 1. 정확성 (Correctness)

무엇을 볼까:
- 변경된 로직이 모든 정상 경로에서 의도한 결과를 내는가 — diff 한 줄 한 줄 머릿속 실행.
- 빈 입력·null·경계값·최대값에서 코드가 어떻게 동작하는지 명시적으로 추적한다.
- 병렬 실행이나 공유 상태(전역 변수·캐시·파일)가 있으면 경쟁 조건 가능성을 확인한다.
- 에러 처리: 예외가 조용히 삼켜지는가, 호출자에게 의미 있는 정보가 전달되는가.

### 2. 보안 (Security)

무엇을 볼까:
- 신뢰할 수 없는 입력(사용자 요청·환경변수·파일)이 경계에서 검증·이스케이프되는가.
- SQL 쿼리·쉘 명령·HTML 출력에 인젝션 가능한 변수가 직접 삽입되지 않는가.
- 시크릿(API 키·패스워드·토큰)이 소스 코드·로그·번들에 포함되지 않는가.
- 새 의존성이 추가됐다면 알려진 취약점 여부(버전 고정 여부 포함)를 확인한다.

### 3. 테스트 품질 (Test Quality)

무엇을 볼까:
- 추가·수정된 테스트가 **행동(행동 = 입력 → 기대 출력)**을 검증하는가, 구현 세부(내부 함수 호출 순서 등)만 보는가.
- 틀린 구현이면 이 테스트가 실제로 실패하는가 — 단언이 없는 테스트는 아무것도 보장하지 않는다.
- 기존 테스트가 수정·삭제됐다면 이유가 정당한가(리팩터링 vs. 게이밍).
- 커버리지 숫자보다 **커버되는 시나리오의 의미**를 본다.

### 4. 스코프 크립 (Scope Creep)

무엇을 볼까:
- diff에 step 요구사항에 없는 파일·기능이 포함되지 않는가.
- "이왕 여기 온 김에" 리팩터링·정리가 섞여 들어오지 않았는가 — 의도 좋아도 스코프 외.
- 설정 파일·빌드 스크립트·문서가 해당 step의 직접 산출물인가, 아니면 편의성 추가인가.
- step 범위 밖 변경이 있으면 `IMPROVE` 사유로 명시하고 별도 step 처리를 요구한다.

### 5. 성능 (Performance)

무엇을 볼까:
- 루프 안에서 DB 쿼리·API 호출·파일 I/O가 반복되는가(N+1 패턴).
- 데이터 크기에 비례해 메모리나 시간이 폭발적으로 증가하는 경우가 있는가.
- 캐시·인덱스·배치 처리가 필요한 곳에서 빠져 있는가.
- 명백한 비효율 외의 조기 최적화는 지적하지 않는다 — 실측 없는 성능 걱정은 과설계다.

## 테스트 게이밍 레드플래그 카탈로그

생성자나 에이전트가 테스트를 통과시키기 위해 테스트 자체를 왜곡하는 패턴이다. diff에서 다음을 보면 즉시 게이밍 의심:

| 레드플래그 | 설명 |
|---|---|
| 단언 없는 테스트 | `test_foo(): pass` — 실행만 되고 아무것도 확인 안 함 |
| `assert True` / `assertEqual(x, x)` | 항상 참인 단언 — 커버리지는 오르나 보장은 0 |
| 테스트를 구현에 맞춰 수정 | 구현이 바뀌자 기대값도 함께 바뀜 — 행동이 아니라 구현을 미러링 |
| `skip` / `xfail` / `.only` 남용 | 실패하는 테스트를 고치지 않고 비활성화 |
| 특정 입력 하드코딩 | 테스트 입력이 구현의 특수 경로와 정확히 일치해 일반 케이스를 검증 못 함 |
| 러너·채점 몽키패치 | 채점 함수를 교체하거나 `exit(0)`으로 조기 탈출 |
| AC 기준 자체 약화 | 통과 조건을 "정답 == output" → "정답 in output"으로 바꾸는 등 |
| 구현 파일·테스트 파일 동시 수정 | 두 파일을 같이 바꿨을 때 기대값 수정 여부를 특히 확인 |

**홀드아웃 테스트**: 스위트에 없는 입력으로 수동 재실행하면 하드코딩이 드러난다 — 의심되면 직접 다른 입력을 넣어 실행하라.

---

## 이 스킬이 잡아야 할 대표 상황

1. **에이전트가 AC를 통과했다고 보고했으나 실제 exit code는 1** — 마지막 줄만 읽고 "통과"라 주장하는 패턴. → `echo $?`를 직접 확인한다.
2. **테스트 파일이 구현과 함께 수정됐고 기대값이 바뀜** — "리팩터링"이라 설명하나 행동 변경임. → 기대값 변경 이유를 요구한다.
3. **새 기능 추가인데 테스트가 전혀 없음** — `git diff --name-only`에 테스트 파일이 없음. → IMPROVE로 반환.
4. **step 범위 밖 파일 수정** — 요구사항에 없는 컴포넌트·설정 파일이 diff에 포함. → 스코프 크립으로 지적.
5. **보안 경계 누락** — API 라우트에서 사용자 입력을 검증 없이 DB 쿼리에 삽입. → 즉시 IMPROVE.
